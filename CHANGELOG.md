## 2026-06-25 [计划] 后端化改造——Supabase + 用户系统

### 背景

当前产品全部数据存储在 localStorage，技术深度不足：无数据库、无用户系统、换浏览器数据丢失。需要后端化改造使产品具备真实可用性。

### 技术选型

**Supabase**（PostgreSQL + Auth + RLS）

选型理由：
- 免费层足够（500MB数据库、50000 MAU）
- 与 Next.js 天然兼容，无需自建后端
- 内置 Auth 支持 GitHub/Google OAuth，无需自己实现
- Row Level Security (RLS) 实现数据隔离，用户只能访问自己的数据

### 实施计划

**Phase 1：数据库Schema设计**
- works 表（作品）
- chapters 表（章节，关联 work）
- worldviews 表（世界观）
- style_presets 表（风格预设）
- characters 表（角色）
- 设计外键关系、RLS策略

**Phase 2：Auth 用户系统**
- 集成 Supabase Auth
- 登录/注册页面（支持 GitHub OAuth）
- 路由保护（未登录跳转登录页）

**Phase 3：数据层迁移**
- 将 localStorage 读写替换为 Supabase 客户端调用
- 保留 localStorage 作为离线缓存/降级方案
- 新用户首次登录时若本地有数据，提示上传到云端

**Phase 4：部署与验证**
- Supabase 项目配置环境变量到 Vercel
- 端到端测试：注册→创建作品→跨设备访问

### 不做的事

- 不做自建后端（Express/Nest等），杀鸡用牛刀
- 不做支付系统（MVP阶段）
- 不做实时协作（超出scope）

## 2026-06-25 世界观推荐条目模板

### 问题

世界观"详细设定"区域完全由用户自行新建条目，用户面对空白状态会茫然无措、不知该写什么。

### 方案

1. **推荐条目按钮** — 详细设定为空时显示「✦ 加载推荐条目」，一键生成4个常见设定分类
2. **按世界类型定制** — 8种世界类型各有不同的推荐条目（如仙侠：江湖格局/武功体系/时代背景/江湖规矩）
3. **引导性placeholder** — 每个条目的编辑框有具体提示问题（如"有哪些门派/势力？"）
4. **保留自由度** — 用户随时可「+ 新增分类」添加任意自定义条目
5. **新建弹窗提示** — 新建世界观时提示"创建后可在世界观页面添加详细设定条目"

### 设计决策

- 推荐条目不是必填，只是辅助性提示，降低用户的空白焦虑
- placeholder用提问形式引导思考，比"请输入内容"有效得多
- 已有同名条目不会重复加载

## 2026-06-25 风格预设精细化——18维度参数 + 范文锚定

### 问题

原有风格系统只有6个粗粒度参数（叙事视角/文风/描写/节奏/情感/敏感度），用户反馈无法精确控制行文风格，AI输出一致性差。

### 方案

在原有基础上增加三个精细控制维度层 + 范文示例：

1. **句式与语言层**（4项）：句子长度（短句/长短交错/长句铺陈）、用词偏好（口语化/书面/半文言）、修辞手法（白描/比喻多/通感/排比/少修辞）、标点习惯
2. **内容比例层**（4项）：对话vs叙述比例、描写类型优先级、内心独白频率、旁白评论频率
3. **章节结构层**（4项）：开篇方式（悬念/对话/环境/倒叙）、转场风格（硬切/渐变/空行留白）、结尾方式（悬念钩子/情感余韵/戛然而止）、每章字数目标
4. **范文示例**（选填）：贴入200-500字喜欢的文字，AI模仿其语感和风格

### 设计决策

- 精细参数在风格预设编辑页以4个独立折叠区呈现，不强制用户全部填写
- 新建作品时的预设弹窗保持简洁（只需名称+风格指令+基础参数），并提示"创建后可在风格预设页面进行更精细调整"
- 6个内置预设全部补充了18维度的推荐默认值
- 范文示例在expand API中作为"严格模仿"指令传递，优先级高于参数描述

## 2026-06-25 章节规划动态化 + 数据一致性 + 功能补全

### 问题

1. **章节规划硬编码** — "让AI帮我规划章节"按钮点击后显示固定的6个假章节（深夜订单/目击/追杀…），无论新建什么书都一样
2. **章节数据丢失** — 新建作品时章节只写入旧格式 `storyforge_chapters`，但编辑器读取新格式 `storyforge_chapters_{workId}`，导致进入编辑器后章节消失只剩一个默认"第一章"
3. **骨架内容为空** — AI规划的每章剧情描述没有保存为骨架内容，用户进入编辑器看到空白，扩写时AI不知道该写什么
4. **导出全文无效** — 按钮没有绑定事件处理函数
5. **模板创建跳转旧作品** — "使用此模板开始创作"只是一个Link到/work/id，打开的是之前的active作品

### 修复

1. **章节规划API化** — 新建 `/api/plan-chapters` 接口，基于用户填写的书名/类型/简介/梗概调用DeepSeek动态生成章节；优化prompt强调不用原著剧情填充if线/同人作品
2. **双格式写入** — `createWork` 同时写入 `storyforge_chapters`（旧格式兼容）和 `storyforge_chapters_{workId}`（新格式）；编辑器的 `loadChapters` 增加旧格式回退
3. **骨架预填** — 新建作品时将每章desc保存为 `storyforge_skeleton_{workId}_{chapterId}`，编辑器加载时骨架已有内容
4. **导出全文实现** — 按章节顺序收集扩写内容（无扩写用骨架），拼接下载为txt文件
5. **模板真实创建** — 按钮改为创建新work+写入章节+设置active，再跳转

### 设计决策

- 章节规划必须动态生成，不能有任何硬编码demo数据残留
- 数据存储需要同时兼容新旧两种格式，过渡期不能丢数据
- AI规划的章节描述即为初始骨架，保证扩写方向与规划一致
- 模板/新建等所有入口都必须真实创建作品数据后再跳转

## 2026-06-25 自定义模式新建作品体验优化

### 问题

新建作品 → 自定义模式中存在三个体验问题：
1. 风格预设按钮点击后弹 `alert()`，无实际选中效果
2. 「+ 新建预设」和「+ 自定义世界观」按钮弹 alert 或跳转到已有作品页面，逻辑不合理
3. 风格预设列表只有 6 个硬编码内置选项，无法选择用户之前创建的自定义预设；世界观也无法复用已有资产

### 修复

1. **风格预设选中** — 点击预设按钮直接高亮选中并持久化到作品数据
2. **结构化弹窗** — 「+ 新建预设」弹出面板含预设名称、风格指令（核心）、6 项参数微调（叙事视角/文风倾向/描写偏好/节奏感/情感浓度/敏感内容尺度）；「+ 新建世界观」弹出面板含 8 种世界类型选择和描述输入
3. **复用已有资产** — 风格预设列表动态读取全部预设（内置 + 用户自定义）；世界观新增「选择已有」按钮，展开全局资产库中所有世界观供选择

### 设计决策

- 新建作品时不应跳转到已有作品的设置页面，所有操作在当前页内完成
- 结构化面板而非纯文本框，降低用户茫然感
- 同时支持「从已有选择」和「从零创建」两条路径

## 2026-06-23 全局资产库架构重构

### 架构决策

**问题**：角色、世界观、风格预设之前要么不持久化（纯内存硬编码），要么全局共享无法区分作品，无法跨书复用。

**新架构**：角色/世界观/预设作为**全局资产**独立存在，每本书通过 ID 引用资产。可 fork 为书内独立副本，修改不影响原始资产。

### 新建数据层模块

1. **`lib/types.ts`** — 统一类型定义（Character, Worldview, Work, Chapter, WorldSection）
2. **`lib/characters.ts`** — 全局角色 CRUD（getCharacters, addCharacter, updateCharacter, deleteCharacter）
3. **`lib/worldviews.ts`** — 全局世界观 CRUD（含 section 管理）
4. **`lib/works.ts`** — 作品管理 + 资产引用/分叉/解析
5. **`lib/migration.ts`** — 旧数据格式自动迁移到新 schema

### 页面改造

6. **角色库页面** ✅ — 持久化到 localStorage，新增"从资产库添加"和"分叉为独立副本"
7. **世界观页面** ✅ — 完全重写，支持创建/引用全局世界观/分叉
8. **风格预设页面** ✅ — 选预设时绑定到当前作品的 stylePresetId
9. **编辑器页面** ✅ — 通过 getWorkCharacters/getWorkWorldview/getWorkStylePreset 解析资产
10. **新建作品** ✅ — 创建时初始化新 schema 字段，世界观存入全局库
11. **首页删除** ✅ — 适配新 schema

### Work 数据模型变更

```
+ characterIds: string[]        // 引用的全局角色ID
+ worldviewId: string | null    // 引用的全局世界观ID
+ stylePresetId: string | null  // 绑定的风格预设ID
+ localCharacters: Character[]  // fork 的角色独立副本
+ localWorldview: Worldview | null // fork 的世界观独立副本
```

## 2026-05-29 新增设计决策：灵感建议功能

### 来源

借鉴酒馆（AI Tavern）扩写后给出 ABCD 选项的机制，但适配 StoryForge 的"用户掌控创意"定位。

### 设计方案

- 在骨架编辑区增加一个"灵感"按钮（按需触发，非自动弹出）
- 用户点击后，AI 根据当前骨架内容 + 角色/世界观设定，生成 3-4 个后续走向建议（每个一两句话，梗概式语气）
- 用户点击某个建议 → 文字追加到骨架编辑区末尾 → 用户可修改后再扩写
- 不点就关掉，继续自己写

### 关键原则

1. 按需触发，不主动打断有灵感的用户
2. 建议追加到骨架（不直接生成成文），保持用户掌控
3. 骨架为空时也能使用（冷启动场景）
4. 建议的语气是梗概式的，与骨架风格一致
